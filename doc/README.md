## 🗂️ ドキュメントの読み方

### リファクタリングについて理解したい場合

```
1. リファクタリング検討 - 意思決定ガイド (決定のための情報)
   ↓
2. リファクタリングサマリー (調査結果の概要)
   ↓
3. リファクタリング計画書 (実装の詳細)
```

### 実装を始める場合

```
1. リファクタリング計画書 (詳細な実施計画を確認)
   ↓
2. チェックリストに従って段階的に実施
   ↓
3. 各ステップでテスト・ビルドを確認
```

### プロジェクト全体の改善点を確認したい場合

```
1. リファクタリングサマリー (最近の分析結果)
   ↓
2. 改善点リスト (その他の改善提案)
   ↓
3. 各種仕様書 (個別機能の詳細)
```

## 📝 ドキュメント作成ガイドライン

新しいドキュメントを追加する際は、以下のガイドラインに従ってください:

### 1. ファイル命名規則

- 小文字とハイフンを使用: `feature-name-specification.md`
- 日本語ドキュメントでも英語ファイル名を推奨
- 内容が分かりやすい名前をつける

### 2. ドキュメント構造

必須セクション:
- タイトル（# 見出し）
- 概要・目的
- 作成日・最終更新日
- 対象読者（必要に応じて）

推奨セクション:
- 背景・課題
- 提案内容
- 期待される効果
- 実装方針
- 参考資料

### 3. 書式

- マークダウン形式（.md）
- 見出しは階層的に使用（#, ##, ###）
- コードブロックには言語指定を追加
- 表やリストを活用して読みやすく
- 絵文字を使って視覚的に分かりやすく

### 4. メンテナンス

- 内容が変更された場合は「最終更新日」を更新
- 古くなった情報には明示的に注記
- 実装完了後はステータスを更新
